# CLAUDE.md — Labalaba

## Overview
Labalaba is a cross-platform **process manager** with a Tauri desktop GUI. It spawns, monitors, and controls arbitrary OS processes (binaries, scripts, Python via runner prefixes) defined as "tasks". Tasks persist to `tasks.yaml` and support start/stop/restart, auto-restart on crash, cron scheduling, dependency ordering with startup delays, admin elevation (Windows UAC), live stdout/stderr log streaming, and per-task CPU/memory stats.

## Architecture
Cargo workspace (`resolver = "2"`) with three members + a SvelteKit frontend:

- `crates/shared` (`labalaba-shared`) — serde DTOs shared by daemon and GUI: `TaskConfig`, `TaskDto`, `TaskRequest`, `TaskStatus`, `TaskStats`, `AppSettings`, `LogEntry`/`LogStream`, `UpdateInfo`, `ApiResponse<T>`. `TaskStatus` serializes `snake_case`; `LogStream` is `lowercase`. **This is the contract** — the TS interfaces in `gui/src/lib/api/client.ts` mirror these by hand; keep them in sync.
- `crates/daemon` (`labalaba-daemon`) — the actual process-manager logic, structured **DDD / hexagonal** (`domain/`, `application/`, `infrastructure/`, `interface/`). Built as **both a lib and a bin**; the **bin is the shipped daemon** the GUI launches (it's bundled as a sidecar), not a dev-only alternative.
- `gui/src-tauri` (`labalaba-gui`, lib name `tauri_app_lib`) — the Tauri 2 shell: window, tray, daemon lifecycle. Depends on `labalaba-daemon` as a library for helpers only (see below).
- `gui/` — SvelteKit 5 + TypeScript + Vite 6, static adapter (`@sveltejs/adapter-static`). The log viewer (`lib/components/LogViewer.svelte`) is **plain Svelte markup**, not a terminal emulator — `@xterm/*` is still declared in `gui/package.json` but imported nowhere in `gui/src` (dead dependency).

**Two processes, one transport.** The product is a GUI **plus** a separate daemon process — not an embedded daemon:

1. **Daemon (`labalaba-daemon`, the process manager):** `crates/daemon/src/main.rs` calls `init_app_state(None, None)` and runs an **axum HTTP + WebSocket server** on `127.0.0.1:{daemon_port}` (default `27015`). Routes in `interface/http/router.rs`; WS log stream at `/ws/logs/{id}`. It owns all task state and outlives the GUI window (that's how auto-restart/cron "survive window close").
2. **GUI (`labalaba-gui`, a thin client):** `gui/src-tauri/src/lib.rs` → `start_or_connect_daemon()` (`commands/daemon.rs`) **spawns the daemon binary as a child process** (or reuses one already on the port, but only if `/api/health` reports the bundled version). It then talks to it over **HTTP + WebSocket** — there is no in-process daemon.

**The frontend is an HTTP client, not a Tauri-command client.** `invoke` is used for exactly one data-layer call: `get_daemon_connection`, which returns `{base_url, ws_url, token}`. Everything else is `fetch(conn.base_url + path)` with a `Bearer` token (`gui/src/lib/api/client.ts`), and logs stream over `WebSocket(${ws_url}/ws/logs/{id}?token=…)` (`websocket.ts`). The only Tauri commands are daemon lifecycle + autostart: `get_daemon_connection`, `daemon_status`, `start_daemon`, `cleanup_daemon`, `set_autostart`, `get_autostart`. **There is no `commands/tasks.rs`** — `gui/src-tauri/src/commands/` holds only `daemon.rs` and `service.rs`.

`gui/src-tauri` still depends on `labalaba-daemon` as a lib, but only for helpers — `data_dir()`, `load_settings()`, `cleanup()`, `stop_running_daemon()`, `daemon_health()`, `infrastructure::autostart` — never to run tasks.

The daemon stays **Tauri-agnostic**. `AppState` still carries two optional `Fn` callbacks (`log_event_callback`, `update_event_callback`, `infrastructure/state.rs`), but **both are always `None` in the shipped product**: `init_app_state`'s only caller is `main.rs`, which passes `(None, None)`. They're a vestigial extension point from the old embedded design — the live path to the UI is the WS stream, not a callback. Don't document or reason about them as if they're wired.

**Key flow:** HTTP/WS route (`interface/http/*_handler.rs`) → application use case (one per file: `CreateTask`, `StartTask`, `StopTask`, `RestartTask`, `EditTask`, `DeleteTask`, `GetLogs`) → domain traits (`TaskRepository`, `ProcessSpawner`) → infrastructure impls (`YamlTaskRepository`, `OsProcessSpawner`, `GithubUpdater`, `LogFileWriter`, `ResourceMonitor`, `cron_scheduler`). `AppState` is `Arc`-shared; runtime status lives in `runtime_states` (in-memory `RwLock<HashMap>`, **not persisted**) while config persists to YAML.

**Auth:** the daemon writes a token to `{data_dir}/daemon.token` on startup (`infrastructure/auth/token.rs`). HTTP routes are behind a `require_token` middleware (`Authorization: Bearer …`); `/api/health` is public; the WS route validates `?token=` inside the handler. Any client — including a manual `curl` — needs that token.

**Updates are split across two mechanisms on purpose — the daemon notifies, the plugin installs.**

1. **Notify (daemon).** `GithubUpdater` (`infrastructure/updater/github_updater.rs`) hits the GitHub releases API, `spawn_update_checker` (`lib.rs`) polls on the `auto_check_updates` / `update_check_interval_hours` settings and fills `state.pending_update`; the frontend polls `GET /api/update/pending` and shows the modal (`gui/src/routes/+page.svelte`). This is what *detects* an update, and it keeps working for a headless daemon with no GUI.
2. **Install (`tauri-plugin-updater`).** Only invoked on-demand when the user clicks Install — it is **not** a second poller. It fetches `latest.json` from `releases/latest/download/latest.json`, verifies a **minisign signature against `plugins.updater.pubkey` in `tauri.conf.json`**, and installs.

The two agree because both resolve to the same git tag. If the plugin's `check()` returns null or throws (a `.deb`/`.rpm` install, Intel Mac, no signed manifest), the modal **falls back to the old "Download Update" link** — that fallback is the intended degradation path, not dead code.

**The updater signing key is a non-recoverable asset.** `TAURI_SIGNING_PRIVATE_KEY` / `TAURI_SIGNING_PRIVATE_KEY_PASSWORD` (repo secrets) sign the artifacts; the matching pubkey is compiled into every installed client. Lose the private key and every installed client loses self-update permanently — the only fix is asking users to reinstall by hand. Rotating the pubkey has the same effect on clients that haven't updated yet.

**Self-update must stop the daemon before installing.** The daemon is a separate process installed *next to* the app and it outlives the window (close-to-tray only hides), so at install time it still holds `labalaba-daemon.exe` open — the very file the installer replaces. `prepare_for_update` (`gui/src-tauri/src/commands/daemon.rs`) does this via the existing `reclaim_port`. The order in `+page.svelte` is **download → `prepare_for_update` → install**, deliberately: `download()` verifies the signature before resolving, so the daemon (and the user's running tasks) only dies once the update is known-good. Stopping first would kill tasks for an update that might never arrive.

**Autostart survives updates via a marker file.** The NSIS pre-uninstall hook (`installer-hooks.nsh`) runs `labalaba-daemon.exe cleanup`, and `cleanup()` removes the autostart entry as well as stopping the daemon — so an upgrade that runs the old uninstaller would silently drop autostart. `prepare_for_update` records the state in `{data_dir}/autostart.pending` and `restore_autostart_after_update` (called from `lib.rs::setup`) re-installs it on the next launch. The marker is only written when autostart *was* enabled, so a user who turned it off never gets it forced back on.

## Key commands
The `Makefile` is the source of truth (it sets `LABALABA_DATA_DIR=$(CURDIR)` for dev). Note the Makefile's `stop`/`release-windows` targets are Windows-oriented.

```bash
make install        # cd gui && npm install
make dev            # Tauri app + hot-reload frontend (sets LABALABA_DATA_DIR to repo root)
make cargo-check    # cargo check -p labalaba-daemon && cargo check -p labalaba-shared
make check          # cd gui && npm run check  (svelte-kit sync + svelte-check, the TS type-check)
make test           # cargo test  (all Rust tests; CI scopes to -p labalaba-daemon -p labalaba-shared)
make build-be       # cargo build -p labalaba-daemon --release  (daemon bin only)
make clean          # cargo clean + rm gui/node_modules gui/.svelte-kit gui/build
```

GUI-only (run inside `gui/`): `npm run dev` (Vite on :1420), `npm run build`, `npm run check`, `npm run tauri <dev|build>`.

Full release bundle: `cd gui && npm run tauri build` (output in `gui/src-tauri/target/release/bundle/`).

## Conventions / things to know
- **Dev profile** (root `Cargo.toml`): `debug = "line-tables-only"`, `incremental = false` — deliberate, for faster Tauri dev builds. Don't "restore" these.
- **Version is synced across 6 files on release** — `gui/src-tauri/tauri.conf.json`, `gui/package.json`, the three `Cargo.toml`s (gui, shared, daemon), and `gui/package-lock.json` (**both** its root `version` and `packages.""."version"`). `release.yml`'s `sed` step rewrites only the **first five** from the git tag, and it does that on a throwaway CI checkout that is never committed back — so the workflow is *not* what keeps the repo consistent, manual bumps are. That gap is exactly why the lockfile silently sat at `1.2.5` from v1.2.6 through v1.2.10. It breaks nothing (`npm ci` only fails on *dependency* drift, not the root version), so nothing will ever fail loudly to remind you. When bumping manually, update all six; `Cargo.lock` follows from any `cargo` command.
- **CI is mostly manual now.** `build.yml` and `test.yml` are `workflow_dispatch`-only (build is Linux-only; the win/mac matrix is commented out). `lint.yml` (`cargo audit` via rustsec) is the **only** workflow that runs on push/PR to `main` (path-filtered to `**/*.rs`, `**/Cargo.toml`, `Cargo.lock`) — and it's advisory (a failure doesn't block merge).
- **`release.yml` triggers on pushing a `v*` tag** — *not* on publishing a GitHub Release. `git push origin v1.2.5` builds and publishes a real release. Its matrix is **5 targets**: win x64/x86, linux amd64/arm64, macOS **Silicon only** — there is no Intel Mac (`x86_64-apple-darwin`) build, and the run still reports success without one.
- **Release is two-phase: draft matrix → `publish` job.** The 5 matrix jobs build as a **draft** (`releaseDraft: true`, `uploadUpdaterJson: false`) and each emits a `fragments/{platform}.json` workflow artifact naming its own updater bundle + signature. A final `publish` job (`needs: build`) merges the 5 fragments into `latest.json`, uploads it, then `gh release edit --draft=false`. Both halves are deliberate: the updater endpoint is `releases/latest/download/latest.json`, so publishing before every target uploads would advertise the release to clients whose platform isn't there yet; and letting all 5 concurrent jobs write `latest.json` themselves is a read-modify-write race that silently drops platforms. The publish job **fails rather than shipping a broken `latest.json`** — it refuses to publish if a platform is missing *or* if any URL names an asset that isn't on the release.
- **An asset's name on the release is not always its local bundle filename.** `tauri-action` renames the macOS bundle on upload to carry its arch (`Labalaba.app.tar.gz` → `Labalaba_aarch64.app.tar.gz`); the other four targets already encode arch/version so they pass through unchanged. v1.2.6 shipped a `latest.json` whose darwin URL 404'd because of exactly this. Each build job therefore resolves its asset name by matching its signature's **content** against the uploaded `.sig` assets — never by assuming the local filename. Don't "simplify" that back into a basename.
- **Data directory** (`crates/daemon/src/lib.rs::data_dir`) — where `tasks.yaml`, `settings.yaml`, `daemon.token`, and `logs/` live. Resolution order: `LABALABA_DATA_DIR` if set and non-empty → else **portable mode** (`{exe_dir}/data`, if `{exe_dir}/labalaba.portable` exists — Windows only, `portable_marker_dir()` is `None` on macOS/Linux since writing next to the exe there breaks the signed bundle or hits a read-only/root-owned path) → else the platform per-user data dir + `"labalaba"` (**Windows `%APPDATA%\labalaba`**, Linux `~/.local/share/labalaba`, macOS `~/Library/Application Support/labalaba`) → else `.`. It is **not** the CWD and **not** the install dir; the Makefile only sets `LABALABA_DATA_DIR` to the repo root for dev. On first run `migrate_legacy_data_dir` best-effort *copies* (never moves/overwrites) `tasks.yaml`/`settings.yaml`/`logs/` from the old CWD location. Toggling portable mode in Settings → Data Location runs the same copy-never-overwrite logic (`copy_data_dir_for_switch`) against the new location and flips the marker; `data_dir()` is deliberately uncached so the switch takes effect on the next call in-process.
- **`config_path`/`log_dir` in settings are relative to the data dir, NOT to the CWD** — always resolve them with `lib.rs::resolve` against `data_dir()`, or better, read the already-resolved path off the component that owns it (e.g. `state.log_writer.log_dir()`). The daemon is spawned by the GUI with **no `current_dir()`**, so it inherits the GUI's CWD — using a raw `"./logs"` against it silently reads the wrong (usually nonexistent) directory. This caused the history fetch in `get_logs.rs` to always return zero lines on a packaged install.
- **Log durability:** `LogFileWriter` batches behind a `BufWriter` and flushes on a policy — 50 lines **or** 500ms. `write`'s own `maybe_flush` only runs *from a write*, so the time half needs `spawn_flusher()` (a 250ms ticker started in `init_app_state`) or a task that logs a burst then goes quiet would leave those lines buffered forever and its `.log` file empty. tokio's `BufWriter` does **not** flush on drop — `close`/`close_all`/rotation must flush explicitly.
- **Runner prefix** (`runner_prefix` on a task, e.g. `"uv run"` or `"python"`) wraps the executable: `exe=uv`, `args=["run", <executable>, ...original args]`. See `resolve_command` in `infrastructure/process/spawner.rs`.
- **On Windows every task inherits all of the daemon's inheritable handles** — and that once bricked the app permanently. `std::process::Command` calls `CreateProcessW` with `bInheritHandles = TRUE` and no `PROC_THREAD_ATTRIBUTE_HANDLE_LIST`, so a child gets a copy of *every* inheritable handle, not just its pipes. mio builds the axum listener with the plain WinSock `socket()` (`std::net` uses `WSASocketW` + `WSA_FLAG_NO_HANDLE_INHERIT`; mio does not), so it starts **inheritable**: a task spawned afterwards held the daemon's `:27015` socket, and when the daemon died without stopping tasks (tray Quit hard-killed it) the port stayed bound with nobody accepting — no later daemon could `bind`, and the GUI never started again until the task was killed by hand. `main.rs` therefore calls `infrastructure::net::disable_handle_inheritance` on the listener **immediately after bind, before any task spawns**; don't move or drop it (`infrastructure/net.rs` has the regression test, which asserts mio's socket still starts inheritable). Accepted per-connection sockets still leak the same way — harmless today (they never block a `bind`, and die with the task), but that's why a dead daemon leaves `CLOSE_WAIT` rows behind. A real fix for the whole class means spawning via raw `CreateProcessW` with a handle allowlist.
- **A port is not a boolean.** `connect()` alone cannot tell "free" from "bound but refusing" — the state a leaked listener leaves behind — and treating that as free is what made the GUI spawn a daemon that could not possibly bind, then report only "daemon did not become reachable". Use `infrastructure::net::probe_port` (`Free` / `Serving` / `Blocked`; `bind` is what separates `Free` from `Blocked`) when deciding whether to spawn. Note a *live* listener that never calls `accept` is `Serving`, not `Blocked` — the kernel completes the handshake into the backlog; `Blocked` in the wild means the owning process is **dead**. `cleanup::reclaim_port_from_orphan_tasks` recovers that state, and it must find the holder via the persisted task pids: the TCP table still names the **dead** daemon that created the socket, never the task actually holding it.
- **Spawning is platform-split** (`spawner.rs`) — don't unify it. Unix runs children on a **PTY** (`spawn_on_pty`) so they line-buffer their logs; Windows spawns onto a **plain pipe** (`spawn_on_pipe`) with `CREATE_NO_WINDOW`, deliberately avoiding ConPTY, and sets `PYTHONUNBUFFERED=1` to keep Python flushing per line without a terminal. A packaged (Nuitka) Python binary was seen dying under ConPTY with `STATUS_DLL_INIT_FAILED` (`0xC0000142`) where a pipe spawn ran it fine — never reduced to a proven mechanism, so treat it as a caution. Either way `ProcessHandle.output` is *one* merged reader (on Windows stdout+stderr share a single `filedescriptor::Pipe`), which is why `start_task.rs` needs no `cfg`. Note `PYTHONUNBUFFERED` only does anything for a **CPython-based** child — it's inert for a Go/Rust/C++ binary, so on Windows whether a non-Python task streams live is entirely down to that child's own buffering (Rust's `std::io::stdout` is a `LineWriter`, so Rust children stream fine; a C program block-buffering a non-tty stdout will not). Before blaming Labalaba for "no logs", check whether the child emits anything at all through a redirected pipe.
- **Crash recovery:** on startup `recover_task_states` re-checks persisted `pids` (`libc::kill(pid,0)` on Unix, `tasklist` on Windows) and marks tasks Running/Crashed. Auto-restart of crashed tasks goes through an `mpsc` channel (`restart_tx` → `restart_loop`) to avoid a recursive-`Send` issue.
- **`TaskRuntimeState` is not persisted** — it's rebuilt in memory each run; only `TaskConfig` (incl. `pids`) lives in YAML.
- **Tests live inline** (`#[cfg(test)] mod tests`) in shared and daemon. `tempfile` (a `crates/daemon` dev-dep) is the workhorse — tests build a real `tempdir` data dir. `mockall` is *declared* in `crates/daemon/Cargo.toml` but currently used nowhere; the domain traits have no generated mocks, so tests that need an `AppState` wire up the real `YamlTaskRepository`/`OsProcessSpawner` against a tempdir instead. No frontend unit tests — `npm run check` (svelte-check) is the frontend gate.
- **The README is accurate on architecture** (it describes the two-process GUI + daemon split over token-authed HTTP/WS on `127.0.0.1`) — trust it over any older "embedded daemon / no HTTP" description you may find elsewhere.
