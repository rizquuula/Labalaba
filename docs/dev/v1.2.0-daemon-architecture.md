# ADR / Plan: v1.2.0 — Daemon-as-a-Service (Option B)

**Status:** Accepted · **Target release:** v1.2.0 · **Date:** 2026-06-13

> Developer-facing architecture decision record and implementation plan. Not end-user documentation (see `docs/en` / `docs/id` for that).

## Context

Today the daemon logic is **embedded in-process** inside the Tauri GUI. The frontend talks to it via Tauri `invoke`/`listen`. Consequences:

- Cron scheduling and auto-restart only live while the GUI window is open — closing the app kills them.
- Already-started tasks *do* survive (they are spawned detached, in their own process group, and are not killed on app exit).
- A second, unused HTTP/WebSocket server already exists in the daemon crate (the standalone `labalaba-daemon` bin), exercising the same use cases over the network.

We want background scheduling/auto-restart that survives the GUI being closed, without maintaining two backends forever.

## Decision

Adopt **Option B: a separate `labalaba-daemon` background process; the Tauri GUI becomes a thin client** that talks to it over `http://127.0.0.1:{port}` + `ws://…/ws/logs/{id}`.

Locked decisions:

1. **Backend model — HTTP only; drop the embedded in-process backend.** Session-only mode spawns the *real* daemon as a child process. One backend, one code path.
2. **Service level — user-level autostart** on all three OSes: `systemd --user` unit (Linux), `launchd` LaunchAgent (macOS), Startup/Run-key + tray (Windows). No elevation; parallel semantics; stops at logout (acceptable per product intent).
3. **Cron format — standard 5-field** (`min hour dom mon dow`), normalized to the crate's 6-field form internally (prepend `0 ` seconds). Fixes the current misleading hint.
4. **The GUI stays a Tauri shell** (not a bare browser) so native file pickers, notifications, "open URL", tray, and autostart keep working. Only the *data layer* moves to HTTP/WS.

### Two launch modes (one binary)

| Mode | Daemon runs as | Cron/auto-restart after GUI close | Started tasks |
|---|---|---|---|
| **Daemon mode** | user-level background service | ✅ keep running | ✅ persist |
| **Session only** | child process of the GUI | ❌ stop (accepted) | ✅ persist (tasks are detached) |

The Tauri shell's new responsibilities: locate the daemon (health-check), start it (service or child) if down, connect, and manage service install/uninstall + single-instance + tray.

## What already exists (reuse as-is)

- HTTP: `GET/POST /api/tasks`, `GET/PUT/DELETE /api/tasks/{id}`, `…/start|stop|restart`, `/api/tasks/{id}/stats`, `/api/stats`, `GET/PUT /api/settings`, `POST /api/update/check`, `GET /api/logs/{id}?lines=N`.
- `WS /ws/logs/{id}` → `LogEntry` JSON via a per-task broadcast channel (cap 1024).
- Auto-restart + crash/PID recovery run inside `init_app_state` → already active in the standalone daemon.
- Daemon binds `127.0.0.1` only.

Key source pointers: daemon bin `crates/daemon/src/main.rs`; router `crates/daemon/src/interface/http/router.rs`; WS `interface/ws/log_handler.rs`; state `infrastructure/state.rs`; init `crates/daemon/src/lib.rs` (`init_app_state`); cron `domain/scheduler/schedule.rs` + `infrastructure/scheduler/cron_scheduler.rs`; GUI client `gui/src/lib/api/client.ts`; GUI events `gui/src/lib/api/websocket.ts`; Tauri commands `gui/src-tauri/src/commands/*`; shell `gui/src-tauri/src/lib.rs`.

## Gaps to close

**Daemon (Rust)**
- D1. **Wire cron**: hold `CronScheduler` on `AppState`; init in `init_app_state`; schedule all scheduled tasks on boot; (un)schedule on create/edit/delete. Accept 5-field cron (normalize to 6).
- D2. **New endpoints**: `GET /api/health`, `GET /api/app/version`, `GET /api/update/pending` (cached background-check result), `POST /api/system/detect-interpreter` (move PATH scan off the Tauri side).
- D3. **Graceful shutdown**: SIGTERM/SIGINT → `state.shutdown()` (flush logs).
- D4. **Token auth** (Phase 2): random token written to `<data dir>/daemon.token` (0600) at startup; GUI sends `Authorization: Bearer`; replace `CorsLayer::permissive()` with the Tauri origin only.
- D5. **Deterministic data dir** (Phase 3): default to a platform per-user dir (`dirs` crate) instead of CWD; `LABALABA_DATA_DIR` still overrides. Service and GUI must resolve the same dir.
- D6. **Daemon single-instance** (Phase 3): lock file / health-check before spawn.

**GUI (TS/Svelte)**
- G1. New HTTP/WS client (`gui/src/lib/api/http.ts`) mirroring the existing `api` surface (15 ops) so stores/components change minimally.
- G2. Real **WebSocket** for `/ws/logs/{id}` (+ reconnect/backoff), replacing `listen('log:{id}')`.
- G3. Replace `listen('update-available')` with polling `GET /api/update/pending`.
- G4. Connection state drives the existing "Cannot connect to daemon" UI; add reconnect.
- G5. Keep Tauri plugins for native bits; `detect_interpreter` calls the daemon endpoint.

**Lifecycle (Tauri shell)**
- L1. Launch flow: resolve data dir → read `daemon_port` (+ token) → health-check → start (service/child) if down → connect.
- L2. Service install/uninstall per OS (user-level).
- L3. `tauri-plugin-single-instance` (focus existing window); optional `tauri-plugin-tray` (Show/Quit, close-to-tray). First-run "Run in background?" stored as a setting (no per-launch modal).

**Packaging & release**
- P1. Bundle daemon as a **Tauri sidecar** (`externalBin`, `name-<target-triple>`), version-locked.
- P2. Update `release.yml` (build daemon per target + sidecar naming) and `Makefile`. Version stays in lockstep across the existing 5 files.

**Migration & docs**
- M1. First-run data migration: if new data dir empty but old CWD location has `tasks.yaml`, copy it over (copy, not move; back up). Running tasks recover via existing PID recovery.
- M2. Docs: scheduling now works (`docs/*/scheduling.md`); add a "Background service / daemon mode" page; fix README HTTP claim + cron hint.

## Phasing (each independently testable)

- **Phase 1 — Daemon completeness** (Rust only): D1 cron wiring + 5-field normalize, D2 endpoints, D3 graceful shutdown. *Test via `curl`/`websocat`; no GUI changes; no auth friction.*
- **Phase 2 — GUI data layer**: G1–G4 + D4 token auth (client sends it). *App works against a hand-started daemon.*
- **Phase 3 — Lifecycle**: L1–L3 + D5 data-dir default + M1 migration + D6 single-instance.
- **Phase 4 — Packaging/release**: P1–P2 + service templates.
- **Phase 5 — Migration polish, docs, QA**: M2, cross-platform test matrix, then the 1.2.0 bump (with pre-bump `/code-review` per workflow).

## Risks / foot-guns

- **Data-dir relocation** (Phase 3) is the highest-risk item — copy + backup, never move; verify before deleting anything.
- **Always-on local API** → token auth (D4) must ship in the same release, not later.
- **Windows elevated (`run_as_admin`) tasks** under an autostart/background context — UAC is awkward; document the limitation.
- **Recovered tasks** still cannot re-stream logs (pre-existing limitation) — document.
- **Cutover** from embedded: mitigated because Phase 2 is testable against a hand-started daemon before lifecycle automation (Phase 3) exists.

## Uninstall cleanup

When the user removes the app, the daemon's autostart entry and running instance must be stopped. User data (`tasks.yaml`, `logs/`) is always preserved — only an explicit in-app action or `labalaba-daemon cleanup --purge` deletes it.

- **`labalaba-daemon cleanup`** — CLI command: stops the running daemon (HTTP shutdown) and removes the user-level autostart entry (systemd unit / launchd plist / Windows Run key). No `--purge` means data is untouched.
- **In-app path** — Settings → "Remove Background Service" calls the same logic for users who never open a terminal.
- **NSIS (Windows .exe installer)** — `gui/src-tauri/installer-hooks.nsh` defines `NSIS_HOOK_PREUNINSTALL`, which runs `"$INSTDIR\labalaba-daemon.exe" cleanup` (via `ExecWait`) before the uninstaller removes files. Wired via `bundle.windows.nsis.installerHooks` in `tauri.conf.json`.
- **Debian (.deb)** — `gui/src-tauri/deb/prerm.sh` is wired via `bundle.linux.deb.preRemoveScript`. **Known limitation:** Debian maintainer scripts run as root, so `labalaba-daemon cleanup` resolves root's data directory (not the installing user's), finds no service entry, and exits cleanly — the script is a no-op for per-user installs. The reliable Linux path is the in-app button or running `labalaba-daemon cleanup` as the correct user before `apt remove`. AppImage has no uninstaller; no hook is possible there.

## Out of scope for v1.2.0

- `depends_on` dependency ordering (still stored-but-inert; separate effort).
- Full system-level services / run-before-login.
- Remote (non-localhost) daemon access / multi-user.
